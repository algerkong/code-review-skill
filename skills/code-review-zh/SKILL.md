# 代码审查系统

一个全面的多维度代码审查技能，生成结构化报告、追踪审查历史，支持从单个文件到整个项目的多种审查范围。

## 核心理念

本技能基于生产级审查系统的方法论（多 Agent 并行分析、基于置信度的过滤、OWASP 对齐的安全检查）。目标是给开发者一份彻底、可操作的报告 — 而不是一堆无关紧要的吹毛求疵。每个发现都应该通过"一个资深工程师会关心这个吗？"的检验才能被纳入报告。

注重**实质胜于形式**：一个关键的逻辑缺陷比缺少末尾换行重要得多。

## 审查模式

本技能支持五种审查模式，根据用户请求判断使用哪种：

| 模式 | 触发语 | 范围 |
|------|--------|------|
| **项目** | "审查整个项目"、"审查代码库" | 整个仓库 — 架构、横切关注点、依赖健康度 |
| **模块** | "审查 auth 模块"、"审查 src/api/" | 某个目录/包 — 内部内聚性、API 接口、模块边界 |
| **文件** | "审查这个文件"、"审查 utils.py" | 单个文件 — 行级细节、逻辑正确性、风格 |
| **差异** | "审查 PR"、"审查这个分支"、"审查变更" | 分支/提交之间的 Git 差异 — 聚焦于变更内容 |
| **Issue** | "审查 issue #42"、"审查登录 bug 的修复" | 围绕特定 Issue 的定向审查 — 代码是否真正解决了问题？ |

如果用户没有指定模式，从上下文推断。如果模糊不清，主动询问。

## 工作流程

每次审查遵循以下流程：

### 第一步：准备 — 了解全局

在阅读任何一行代码之前，先收集上下文：

1. **检查历史报告**：读取 `.review/history.json`（如果存在）以及该范围最近的报告。这一步至关重要 — 它告诉你之前标记了什么、哪些已解决、哪些出现了回归。

2. **理解项目**：查看项目结构、`README`、配置文件（`package.json`、`pyproject.toml`、`Cargo.toml`、`go.mod` 等）以了解技术栈、框架和约定。

3. **差异模式**：运行 `git log` 和 `git diff` 了解变更内容和原因。如果有 PR 描述或提交信息，也要阅读。

4. **Issue 模式**：先理解 Issue 的上下文 — bug/功能是什么、预期行为是什么，然后追踪相关代码路径。

### 第二步：多维度分析

从以下维度审查代码（根据审查模式调整深度）：

#### 维度 1：逻辑正确性
- 代码是否按预期工作？
- 边界情况：null/undefined、空集合、边界值、溢出
- 并发代码中的竞态条件
- 错误处理：失败是否被捕获、正确传播、可恢复？
- 状态管理：系统能否进入无效状态？

#### 维度 2：安全性（OWASP 对齐）
- **注入**：SQL、XSS、命令注入、路径遍历 — 输入是否经过验证？
- **认证授权**：每个路径都检查了认证和授权？
- **数据泄露**：敏感数据是否被记录在日志中、在错误中泄露、或未加密传输？
- **依赖项**：第三方包中是否有已知漏洞？
- **密钥泄露**：硬编码的凭证、API 密钥、Token？
- **CSRF/SSRF**：状态变更请求是否受到保护？

#### 维度 3：性能
- N+1 查询、无界循环、热路径中的不必要分配
- 资源泄漏：未关闭的连接、文件句柄、流
- 算法复杂度与预期数据量的匹配
- 缓存机会、不必要的重复计算
- 应该异步化的阻塞调用

#### 维度 4：架构与设计
- 变更是否符合现有架构还是与之冲突？
- 耦合性：是否在模块之间创建了隐式依赖？
- 抽象：层次是否合适，是否存在泄漏？
- SOLID 原则：单一职责、依赖反转
- API 设计：公共接口是否清晰且最小化？

#### 维度 5：可维护性与可读性
- 命名：名称是否揭示意图？
- 函数：小而专一，处于同一抽象层次？
- 注释：解释"为什么"而不是"做什么"？死代码是否已删除？
- 测试覆盖：行为关键路径是否有测试？
- 与项目约定的一致性

#### 维度 6：项目级关注点（仅项目模式）
- 依赖健康度：过时的包、许可证冲突
- 配置规范：环境变量、密钥管理
- CI/CD：流水线是否正确配置？
- 文档：README 准确性、API 文档
- 代码组织：项目结构是否合理且易于导航？

### 第三步：严重级别分类

每个发现都标注严重级别。保持诚实和校准 — 严重级别的通胀会摧毁信任。

| 级别 | 图标 | 含义 | 所需操作 |
|------|------|------|----------|
| **严重** | 🔴 | 安全漏洞、数据丢失风险、生产环境崩溃 | 合并/部署前必须修复 |
| **高** | 🟠 | 重要 bug、真实负载下的性能问题、认证缺口 | 应尽快修复 |
| **中** | 🟡 | 逻辑异味、缺少边界处理、糟糕的错误信息 | 方便时修复 |
| **低** | 🔵 | 风格不一致、命名改进、小型重构机会 | 锦上添花 |
| **信息** | ⚪ | 观察、对良好模式的表扬、架构注记 | 无需操作 |

**置信度评分**：对每个发现，内部评估置信度（0-100）。只纳入置信度 ≥ 70 的发现。如果是猜测，明确说明。误报比遗漏更快地侵蚀信任。

### 第四步：生成报告

报告存放在项目根目录的 `.review/` 目录中，不存在则创建。

**报告命名规范**：
```
.review/
├── history.json                          # 审查历史索引
├── reports/
│   ├── 2026-03-12_project_full.md       # 项目审查
│   ├── 2026-03-12_module_src-api.md     # 模块审查（路径编码）
│   ├── 2026-03-12_file_utils-py.md      # 文件审查
│   ├── 2026-03-12_diff_feature-auth.md  # 差异/PR 审查（分支名）
│   └── 2026-03-12_issue_42.md           # Issue 审查
└── snapshots/                            # （可选）用于差异追踪的代码快照
```

**报告格式** — 使用 `assets/report_template.md` 中的模板。关键章节：

```markdown
# 代码审查报告

## 元信息
- **日期**: YYYY-MM-DD HH:MM
- **审查者**: Claude（AI 辅助审查）
- **模式**: 项目 | 模块 | 文件 | 差异 | Issue
- **范围**: [审查了什么 — 路径、分支、Issue 编号]
- **上次审查**: [上次报告链接或"首次审查"]
- **技术栈**: [检测到的语言、框架]

## 摘要
[2-3 句话：总体健康评估、最关键发现、核心建议]

## 统计数据
- 分析文件数：N
- 总发现数：N（严重：N，高：N，中：N，低：N，信息：N）
- 与上次审查对比：+N 新增 / -N 已解决 / N 持续存在

## 严重和高级发现
[详细发现：文件、行号、描述、建议修复]

## 中级发现
[...]

## 低级和信息级发现
[...]

## 与上次审查的对比
### 已解决的问题（来自上次报告）
### 新增的问题（上次没有的）
### 持续存在的问题（仍然存在的）

## 建议
### 立即行动（本迭代）
### 短期改进（未来 2 个迭代）
### 长期架构笔记

## 附录
### 已审查文件列表
### 审查维度覆盖情况
```

### 第五步：更新历史

生成报告后，更新 `.review/history.json`：

```json
{
  "project": "项目名称",
  "reviews": [
    {
      "id": "review-2026-03-12-001",
      "date": "2026-03-12T14:30:00Z",
      "mode": "project",
      "scope": "完整项目",
      "report_path": "reports/2026-03-12_project_full.md",
      "summary": {
        "total_findings": 15,
        "critical": 1,
        "high": 3,
        "medium": 6,
        "low": 4,
        "info": 1
      },
      "resolved_from_previous": 3,
      "new_findings": 5,
      "recurring_findings": 7
    }
  ]
}
```

## 各模式详细指令

### 项目审查
1. 先进行高层扫描：`find` 文件类型，检查目录结构，读取配置文件
2. 识别最关键的模块（认证、支付、数据访问） — 深入审查这些
3. 抽样其他模块 — 不需要读每个文件，但要覆盖足够多以发现模式
4. 检查横切关注点：日志、错误处理模式、配置管理
5. 审查依赖清单中过时/有漏洞的包
6. 如果有子代理：为不同维度启动并行审查

### 模块审查
1. 映射模块边界：公共 API 是什么，内部是什么？
2. 审查模块中的所有文件
3. 检查内部内聚性 — 所有文件都属于这里吗？
4. 检查耦合性 — 这个模块依赖什么，什么依赖它？
5. 评估模块的测试覆盖率

### 文件审查
1. 阅读整个文件
2. 理解它在更广泛代码库中的角色（读取导入，查找调用者）
3. 逐行分析所有维度
4. 这是最详细的模式 — 包括低级发现

### 差异审查（PR/分支）
1. 获取差异：`git diff <base>...<head>` 或 `git diff <base> <head>`
2. 阅读 PR 描述/提交信息了解意图
3. 对于每个变更的文件，理解变更周围的上下文（不仅仅是 diff 行）
4. 审查重点：这个变更是否完成了它声称的功能？是否破坏了什么？是否引入了风险？
5. 检查：新行为是否有测试？
6. 对比 `.review/` 历史中受影响文件的记录

### Issue 审查
1. 理解 Issue（阅读描述、评论、关联的 PR）
2. 识别与 Issue 相关的代码路径
3. 深入审查这些特定路径
4. 评估：当前代码（或提议的修复）是否真正解决了根本原因？
5. 检查其他地方是否有类似模式可能存在相同问题

## 处理不同编程语言

审查清单普遍适用，但根据语言调整重点：

- **JavaScript/TypeScript**：重点关注 async/await 陷阱、类型安全（TS）、原型污染、XSS
- **Python**：重点关注类型提示使用、导入组织、Django/Flask 安全、资源管理
- **Java/Kotlin**：重点关注空安全、并发、Spring 安全配置、内存泄漏
- **Go**：重点关注错误处理模式、goroutine 泄漏、接口设计
- **Rust**：重点关注 unsafe 块、生命周期问题、Result 错误处理
- **C/C++**：重点关注内存安全、缓冲区溢出、RAII 模式、未定义行为
- **Ruby**：重点关注 Rails 安全、元编程复杂性、ActiveRecord 的 SQL 注入
- **PHP**：重点关注注入漏洞、类型转换陷阱、认证模式

## 使用 Git

审查常用的 git 命令：

```bash
# 分支差异
git diff main...feature-branch
git log main..feature-branch --oneline

# 最近的变更
git log --oneline -20
git diff HEAD~5..HEAD

# 文件历史
git log --follow -p -- path/to/file

# 查找相关变更
git log --all --grep="issue-keyword"

# 检查合并冲突
git merge-tree $(git merge-base main feature) main feature
```

## 报告质量标准

报告本身是一个交付物 — 应该专业且有用：

- 每个发现包含：位置（文件:行号）、描述、严重级别、建议修复方向
- 摘要对忙碌的技术负责人真正有用（他们能从中判断"合并还是不合并？"）
- 与前次报告的对比要具体（"3月5日审查中的 #3 问题现已解决"）
- 建议要有优先级且可操作，不要含糊（"提高代码质量"没有用）
- 不确定的事情要说明 — "这个模式*可能*在高并发下导致问题，但需要看到实际的负载情况才能确定"

## 重要提醒

- **务必先检查历史报告**。这是审查系统随时间推移创造价值的关键 — 它建立了机构记忆。跳过这一步意味着你会重复报告已解决的问题并遗漏回归。
- **诚实校准严重级别**。如果一切都是"严重"，那就什么都不是。正确识别 3 个真实问题的审查比标记 30 个误报的审查更有价值。
- **保持建设性**。每个批评都应该附带建议的前进方向。"这是错误的"没有帮助。"这里容易受到 SQL 注入攻击 — 使用参数化查询，像这样：[示例]"才有帮助。
- **表扬好的模式**。使用信息级发现来指出写得好的代码。这使审查感觉平衡，并鼓励良好实践。
