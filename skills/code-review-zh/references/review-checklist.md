# 代码审查清单 — 40 个问题

需要进行全面审查时阅读此参考文件。按维度组织，每个问题包含检查要点和常见陷阱。

---

## 维度 1：逻辑正确性（问题 1-10）

### 1. 代码是否正确实现了所有声明的需求？
- 将每个需求追溯到具体的代码路径
- 检查是否满足了验收标准，而不仅仅是正常路径

### 2. 循环边界是否正确？
- 迭代中的差一错误（off-by-one）
- 无限循环可能性（缺少跳出条件、不正确的收敛）
- 空集合处理（循环体是否假设 ≥1 个元素？）

### 3. 如果变量或对象在运行时为 null/undefined 会发生什么？
- 追踪可空引用的调用链
- 检查是否需要可选链（JS/TS 的 `?.`，Kotlin 的 `?`）
- 解引用前的空值检查

### 4. 所有用户输入是否经过全面验证？
- 类型验证、范围检查、格式验证
- 字符串和集合的长度限制
- 白名单 vs 黑名单方式（优先白名单）

### 5. 代码做了哪些隐式假设？
- 排序假设（事件按顺序到达）
- 唯一性假设（ID、用户名）
- 可用性假设（外部服务始终响应）

### 6. 线程交错是否可能导致意外行为？
- 多线程访问的共享可变状态
- 没有同步的检查-然后-操作模式
- 原子操作假设

### 7. 共享资源是否有适当的同步保护？
- 具有正确隔离级别的数据库事务
- 需要时使用分布式锁
- 锁排序中的死锁可能性

### 8. 操作中途失败会发生什么？
- 错误前的部分状态变更
- 清理/回滚逻辑
- 可重试操作的幂等性

### 9. 代码如何处理时间相关的边界情况？
- 时区处理（UTC vs 本地时间）
- 夏令时转换
- 分布式系统中的时钟偏差
- 日期运算（不同天数的月份、闰年）

### 10. 系统是否可能通过意外的操作序列进入无效状态？
- 状态机完整性（所有转换已定义）
- 并发请求处理（重复提交、竞态条件）
- 初始化顺序依赖

## 维度 2：安全性 — OWASP 对齐（问题 11-20）

### 11. 输入在用于 SQL、OS 或 LDAP 操作前是否经过验证和净化？
- 参数化查询（非字符串拼接）
- OS 命令参数转义或完全避免
- LDAP 过滤器编码

### 12. 密码/凭证是否使用强哈希存储？
- 可接受：bcrypt、Argon2、PBKDF2（足够的轮数）
- 不可接受：MD5、SHA-1、无盐的 SHA-256、明文
- 每个凭证的盐值唯一性

### 13. 每个请求（包括直接对象引用）都检查了授权吗？
- IDOR（不安全的直接对象引用）防护
- 服务端授权（不依赖客户端检查）
- 中间件/装饰器对认证的一致使用

### 14. 敏感数据在传输中（TLS 1.2+）和静止时是否加密？
- HTTPS 强制执行（无混合内容）
- PII 的数据库加密
- 加密密钥管理（非硬编码）

### 15. 安全头是否已配置？
- Content-Security-Policy (CSP)
- Strict-Transport-Security (HSTS)
- X-Frame-Options, X-Content-Type-Options
- Referrer-Policy

### 16. 用户数据在渲染上下文中是否正确编码/转义？
- HTML 输出的 HTML 实体编码
- 内联脚本的 JavaScript 字符串转义
- 样式上下文的 CSS 编码
- URL 参数的 URL 编码

### 17. 外部依赖是否经过已知漏洞扫描？
- `npm audit`、`pip-audit`、`cargo audit`、Dependabot、Snyk
- 依赖固定（存在并已提交锁文件）
- 许可证兼容性

### 18. 安全相关事件是否有充分的日志记录？
- 认证尝试（成功和失败）
- 授权失败
- 输入验证失败
- 日志中不包含敏感数据（密码、Token、PII）

### 19. 用户提供的 URL 是否通过白名单验证以防止 SSRF？
- URL 协议限制（仅 http/https）
- 域名白名单
- 内部网络范围屏蔽（169.254.x.x、10.x.x.x 等）

### 20. 每个状态变更请求是否有反 CSRF Token？
- Token 生成和验证
- SameSite Cookie 属性
- 双重提交 Cookie 模式（如适用）

## 维度 3：性能（问题 21-30）

### 21. 代码是否在循环内执行查询？
- 经典的 N+1 问题
- 查找循环中 ORM 的延迟加载
- 批量或预加载替代方案

### 22. 多个数据库调用能否合并为一个批量或 JOIN 查询？
- 可以是单个查询的顺序读取
- 可以批量化的独立 insert/update 调用
- 事务范围（太宽 = 争用，太窄 = 不一致）

### 23. 在每个代码路径中，所有分配的资源是否都被释放？
- try-finally / using / defer / with 模式
- 连接池归还
- 文件句柄和流的关闭

### 24. 是否有内存集合无限增长？
- 没有淘汰策略的缓存（LRU、TTL）
- 事件监听器累积
- 日志缓冲区

### 25. 热路径中是否存在不必要的对象分配？
- 紧密循环内的对象创建
- 字符串拼接 vs StringBuilder
- 数值密集代码中的自动装箱（Java）

### 26. 算法复杂度对于预期数据量是否可接受？
- 在潜在大型数据集上的 O(n²) 或更差
- 大表上的无索引数据库查询
- 没有深度限制的递归算法

### 27. 重复计算能否被缓存？
- 使用相同参数调用的纯函数
- 渲染循环中的昂贵计算（React useMemo）
- 结果稳定的数据库查询

### 28. 线性搜索能否替换为索引查找？
- Map/Set 能实现 O(1) 的地方使用 Array.find()
- 有索引或应该有索引的数据库全表扫描
- 对同一集合的重复过滤

### 29. 同步瓶颈是否限制了水平扩展？
- 全局锁、单线程队列
- 数据库行级 vs 表级锁
- 阻止分片的共享可变状态

### 30. 代码是否进行了可以异步化的阻塞调用？
- 异步运行时中的同步 HTTP 请求
- 阻塞事件循环的文件 I/O
- 不需要顺序等待的数据库查询

## 维度 4：可维护性与可读性（问题 31-40）

### 31. 这个变更是否改善了整体代码健康度（即使不完美）？
- 童子军规则 — 让营地比你来时更干净
- 渐进改善优于追求完美
- 不要因为要求卓越而阻止好的变更

### 32. 名称是否有意义且揭示意图？
- 变量名描述内容而非类型（`userAge` 而非 `int1`）
- 函数名描述行为（`calculateShippingCost` 而非 `process`）
- 布尔值名称自然可读（`isEnabled`、`hasPermission`）

### 33. 函数是否小而专一，处于同一抽象层次？
- 做太多事情的函数（> 1 个明确职责）
- 混合的抽象层次（HTTP 解析紧邻业务逻辑）
- 提取方法的机会

### 34. 函数名是否无需查看实现就能描述行为？
- 名称描述"做什么"而非"怎么做"
- 实现细节不泄漏到名称中
- 代码库中一致的命名模式

### 35. 注释是否解释"为什么"而非"做什么"？死代码是否已删除？
- 解释业务规则或非显而易见的决策的注释
- 应该删除的注释代码（它在 git 历史中）
- 带跟踪的 TODO/FIXME/HACK 注释（Issue 编号）

### 36. 相关代码是否放在一起且有清晰的分隔？
- 内聚：相关逻辑在同一模块中
- 分隔：不相关的关注点在不同模块中
- 垂直排序：调用者在被调用者之上，或公开在私有之上

### 37. 这个变更是否减少了耦合并增加了灵活性？
- 依赖注入优于硬依赖
- 使用接口/协议实现多态
- 配置优于硬编码值

### 38. 错误是否被明确处理并提供有用、安全的消息？
- 面向用户的错误：有帮助但不暴露内部细节
- 面向开发者的错误：堆栈跟踪、上下文、关联 ID
- 没有吞掉的异常（空 catch 块）

### 39. 变更是否包含验证行为的测试？
- 新逻辑的单元测试
- 新集成的集成测试
- 边界情况覆盖（不仅是正常路径）
- 测试质量：测试行为还是实现细节？

### 40. 代码是否一致地遵循已建立的风格指南？
- 项目自身的风格指南（如果存在）
- 语言惯例和约定
- 与代码库其余部分的一致模式
